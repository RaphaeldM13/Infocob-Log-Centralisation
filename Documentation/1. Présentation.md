# 1. Introduction

  Le système présenté dans cette documentation permet la collecte et la centralisation de logs provenant de machines Apache, Linux, Windows et Docker.
  Les logs sont collectés par l’agent Grafana Alloy, qui assure leur traitement et leur enrichissement, avant de les transmettre à une plateforme de centralisation basée sur Grafana Loki.
  Cette dernière est déployée sous forme de conteneur Docker et utilise SeaweedFS comme solution de stockage objet.

# 2. Architecture globale

  Le système repose sur une architecture de collecte distribuée des logs.

  Chaque machine à superviser (serveur Apache, système Linux/Windows, réseau docker) embarque un agent Grafana Alloy chargé de collecter et transmettre les logs. Alloy assure la lecture des journaux locaux, leur enrichissement (labels) et leur envoi vers une plateforme centralisée.

  Les logs sont ensuite envoyés via HTTP vers Grafana Loki, déployé dans un conteneur Docker. Loki centralise et stocke les données, en s’appuyant sur SeaweedFS pour le stockage objet.
  Le flux de données est le suivant :
    ```
    Sources → Alloy → Gateway (Nginx) → Loki (read/write/backend) → SeaweedFS
    ```
  Cette architecture permet une solution scalable, flexible et facilement déployable sur plusieurs machines.

# 3. Présentation des composants
  ## Grafana

  Grafana est une plateforme open source de visualisation et d'analyse de données,
  développée par Grafana Labs. Elle permet d'interroger, d'afficher et d'alerter
  sur des données issues de multiples sources via des dashboards personnalisables.

  Dans ce projet, Grafana constitue l'interface d'exploitation des logs centralisés
  par Loki. Il est connecté à Loki comme datasource et expose :
  - les logs bruts filtrables par labels ou par contenu textuel
  - des visualisations de répartition par OS, type de log, statut HTTP
  - un système d'alertes notifiant par email en cas d'anomalie détectée

  L'utilisation de Grafana est pertinente dans ce projet pour plusieurs raisons :

  - <ins>Intégration native avec Loki</ins> : datasource officielle, requêtes LogQL
    directement dans l'interface
  - <ins>Alerting intégré</ins> : les règles d'alerte et contact points sont
    provisionnés automatiquement au démarrage
  - <ins>Multi-sources</ins> : si l'infrastructure évolue vers des métriques ou
    des traces, Grafana peut les unifier dans les mêmes dashboards sans changer d'outil

  ## Alloy
  Grafana Alloy est un agent de collecte de télémétrie permettant de récupérer, traiter et transmettre des données telles que des logs, métriques ou traces. Dans ce projet, il est utilisé comme agent de collecte de logs déployé sur chaque machine à superviser.

  Alloy permet de lire différentes sources de logs (fichiers système, journaux applicatifs, conteneurs Docker), puis d’appliquer des traitements tels que le parsing et l’ajout de labels, avant d’envoyer les données vers une plateforme centralisée comme Grafana Loki.

  Dans ce projet, Grafana Alloy agit comme agent de collecte de logs déployé sur chaque machine à superviser.
    Il est chargé de récupérer les logs locaux (notamment ceux du serveur Apache), de les enrichir (ajout de labels) puis de les transmettre à Grafana Loki pour leur centralisation et leur exploitation.
    Alloy constitue ainsi l’élément intermédiaire essentiel entre les sources de logs et la plateforme de stockage.

  L’utilisation de Grafana Alloy est particulièrement pertinente dans ce projet pour plusieurs raisons :

  - <ins>Flexibilité</ins> : prise en charge de multiples sources de logs (Apache, système, Docker)
  - <ins>Centralisation simplifiée</ins> : envoi direct vers Loki via HTTP
  - <ins>Architecture distribuée</ins> : un agent par machine, facilitant le passage à l’échelle
  - <ins>Capacités de traitement</ins> : enrichissement et structuration des logs avant ingestion

    Ainsi, Alloy constitue un composant clé permettant d’unifier et de standardiser la collecte des logs au sein de l’infrastructure.
    
  ## Gateway (Nginx)

  Un reverse proxy Nginx fait office de point d'entrée unique pour Loki.
  Il écoute sur le port 3100 et route les requêtes vers les composants
  appropriés selon l'endpoint appelé :
  - les requêtes d'ingestion (`/loki/api/v1/push`) sont routées vers `loki-write`
  - les requêtes de lecture (`/loki/api/.*`) sont routées vers `loki-read`

  ## Loki
  Grafana Loki est le composant central du système de centralisation des logs. Il est responsable de la réception, du stockage et de la mise à disposition des logs provenant des différentes machines supervisées via les agents Grafana Alloy.

  Les logs sont envoyés à Loki via une API HTTP d’ingestion exposée sur l’endpoint suivant :
      ```
      http://<IP_LOKI_HOST>:3100/loki/api/v1/push
      ```
  Loki ne se contente pas de stocker les logs, il les indexe principalement via leurs labels, ce qui permet des requêtes rapides et efficaces dans Grafana sans nécessiter une indexation complète du contenu.

  Loki est ici déployé en mode distribué dans un docker avec trois composants distincts :

  | Composant | Rôle |
  |---|---|
  | `loki-write` | Ingestion des logs entrants |
  | `loki-read` | Traitement des requêtes de lecture (LogQL) |
  | `loki-backend` | Compaction et gestion des règles ruler |

  ## SeaweedFS

  SeaweedFS est un système de stockage objet open source exposant une API
  compatible S3. Dans ce projet, il constitue la couche de persistance de
  Loki : les logs ingérés sont stockés dans deux buckets (`loki-data` et
  `loki-ruler`).

  Il est déployé en mode `server` consolidé (master + volume + filer + S3)
  dans la même stack Docker que Loki, ce qui évite toute dépendance à un
  service de stockage cloud externe.

# 4. Prérequis

  ## Versions requises

  | Composant      | Version minimale | Notes                           |
  |----------------|-----------------|----------------------------------|
  | Docker Engine  | 20.10           | 24.x recommandé                  |
  | Docker Compose | v2              | Intégré à Docker Engine (plugin) |

  > ⚠️ Docker Compose **v1** (`docker-compose`) n'est plus maintenu depuis 2023.  
  > La commande à utiliser est `docker compose` (sans tiret).

  ## Permissions

  ### Accès au daemon Docker

  Par défaut, seul `root` peut interagir avec le daemon Docker.  
  Pour autoriser un utilisateur non-root à exécuter des commandes Docker :

  1. Vérifier que le groupe `docker` existe :
     ```
     grep docker /etc/group
     ```
  3. Le créer si absent :
     ```
     sudo groupadd docker
     ```
  5. Ajouter l'utilisateur au groupe :
     ```
     sudo usermod -aG docker $USER
     ```
  7. Appliquer sans déconnecter la session :
     ```
     newgrp docker
     ```
  > **Avertissement sécurité** : L'appartenance au groupe `docker` est équivalente  
  > à des droits `root` sur la machine hôte. Un membre de ce groupe peut monter  
  > n'importe quel chemin du système hôte dans un conteneur.  
  > N'ajouter que les utilisateurs qui en ont strictement besoin.  
  > Source : [Docker Docs — Linux post-install](https://docs.docker.com/engine/install/linux-postinstall/)

  ### Vérification du socket Docker

  Alloy accède au daemon via `/var/run/docker.sock`.  
  Vérifier que le socket appartient bien à `root:docker` :
  ```
    ls -l /var/run/docker.sock
  ```
  La sortie attendue est :
  ```
    srw-rw---- 1 root docker ...
  ```
  Si ce n'est pas le cas :
  ```
  sudo chown root:docker /var/run/docker.sock
  sudo chmod 660 /var/run/docker.sock
  ```
