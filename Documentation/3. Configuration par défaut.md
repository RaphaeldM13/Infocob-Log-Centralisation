# Configuration par défaut
La configuration suivante correspond à l’état du déploiement après l’installation de Loki et le déploiement d’Alloy sur au moins une machine.

## Compatibilité des environnements :
Actuellement, alloy peut être déployé sur trois environnements :
- Linux : Récupération et tri des logs système et applicatifs.
- Windows : Récupération et tri des logs système, applicatifs et de sécurité.
- Docker : Récupération des logs des containers présents sur le réseau Docker où Alloy est déployé.

Les logs Docker pouvant avoir des formats très différents selon les containers utilisés, aucun pipeline de parsing générique n’est actuellement appliqué à cette source.

## Traitement des logs :
  ### Labels communs
  Tous les logs parsés possèdent les labels suivants :
  | label | description |
  |-------|-------------|
  | architecture | architecture de la machine source |
  | filename | fichier source du log |
  | hostname | nom de la machine source/ID du conteneur si la source est un Docker |
  | levelText | niveau du log |
  | os | système d’exploitation de la machine source |
  | source | origine du log |
  | version | version de alloy présente |
  > flename n'est pas disponible sous Windows, les logs étant récupérés directement via windowsevent

  En fonction de l'environnement, les logs peuvent obtenir des labels spécifiques.

  ### Labels spécifiques
  Certains environnements ajoutent des labels supplémentaires.
  
  #### Apache
  | label | description |
  |-------|-------------|
  | http_class | classe du code HTTP retourné |
  | ip | adresse IP ayant effectué la requête |
  | method | méthode HTTP utilisée |
  | path | endpoint ou ressource appelée |
  | protocol | protocole HTTP utilisé |
  | status | code HTTP retourné |
  #### Docker
  
  | label | description |
  |---|---|
  | service_name | nom du container ayant généré le log |
  > Exception : `unknown_service` désigne les logs provenant des agents Alloy
  > déployés sur des machines externes (Linux, Windows, Apache), qui ne
  > possèdent pas de label `service_name` explicite.

## Alertes configurées :
Les alertes actuellement configurées sont réparties en trois catégories.
  ### Attack :
  - Bruteforce : Déclenchée lorsqu’une machine produit un nombre anormalement élevé de logs liés à des échecs d’authentification.
  ### Error :
  - Server-error : Déclenchée lorsqu’une machine génère un volume important d’erreurs serveur HTTP (5xx).
  - Client-error : Déclenchée lorsqu’une même adresse IP produit un volume important d’erreurs client HTTP (4xx).
  ### Silent :
  - NoData : Déclenchée lorsqu’une machine cesse d’envoyer des logs pendant une période définie.
  ### Ajustement des seuils
  Les seuils de déclenchement des alertes devront être adaptés en fonction :

  - du nombre de machines supervisées
  - du volume de logs généré
  - du comportement attendu des applications

  Un mauvais dimensionnement peut entraîner :

  - des déclenchements excessifs
  - des faux positifs
  - la non-détection d’incidents réels

## Dashboard

Le provisioning automatique des dashboards n’étant pas encore implémenté, le [fichier .JSON](/Config/provisioning/dashboards/Infocob_Logs_Dashboard.JSON) du dashboard doit être téléchargé puis importé manuellement depuis l’interface Grafana via :

`Dashboards > Import`

Le dashboard fourni dans cette documentation à titre d’exemple contient 7 panneaux :

les 6 premiers affichent les logs triés selon les labels qui leur sont associés
le dernier affiche un graphique représentant la répartition des logs par système d’exploitation sur les dernières 24 heures

Deux variables sont définies dans le dashboard fourni :
### Recherche 
- type Text

  Permet de filtrer les logs par contenu textuel. La valeur saisie est injectée dans les requêtes via `|= \$Recherche`. Ce type a été retenu pour sa simplicité d'usage : l'utilisateur saisit librement un extrait de log sans contrainte de valeur prédéfinie.
  
### search_label 
- type Ad hoc filters

  Permet de filtrer les logs par labels Loki directement depuis le dashboard, sans modifier les requêtes des panneaux. Les clés et valeurs disponibles sont dynamiquement proposées à partir des labels indexés dans Loki.

Par défaut :

 - les logs affichés couvrent les dernières 24 heures
 - le rafraîchissement du dashboard est automatique
 - la fréquence de rafraîchissement s’adapte au volume de logs reçu
