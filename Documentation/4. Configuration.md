# 1. Configuration de Grafana
Grafana est conçue pour être principalement configurée via son interface utilisateur. La modification manuelle des fichiers de configuration est réservée aux paramètres devant persister à travers les déploiements : SMTP, provisioning, datasources, alerting.
  ## 1.1 SMTP
  
  La configuration SMTP est centralisée dans le fichier [.env](Config/.env). Les variables suivantes doivent être renseignées avant le démarrage de la stack :
  | variables | effet |
  |---|---|
  | `GF_SECURITY_ADMIN_PASSWORD` | Mot de passe du compte administrateur Grafana |
  | `GF_SMTP_HOST` | Serveur SMTP (ex : smtp.gmail.com) |
  | `GF_SMTP_FROM_ADDRESS` | Adresse affichée comme expéditeur des notifications |
  | `GF_SMTP_PASSWORD` | Mot de passe d'application généré par le fournisseur |
  | `GF_SMTP_USER` | Adresse utilisée pour l'authentification SMTP |
    
  ## 1.2 Datasources

  Les datasources provisionnées sont déclarées dans [datasource.yaml](Config/provisioning/datasources/datasources.yaml). Lorsque `editable: true` est défini, elles peuvent également être modifiées via l'interface Grafana (Connections > Data sources).

  ## 1.3 Dashboards

  La configuration des dashboards s'effectue uniquement via l'interface Grafana. Le provisioning automatique via fichier JSON n'est pas supporté pour le format API v2 utilisé dans ce dépôt — l'import manuel est la seule méthode disponible.
  
  ### Importer un dashboard existant
  `Dashboards > New > Import` —> charger le fichier JSON ou coller son contenu.
  ### Créer un panneau
  Depuis un dashboard : 
    Cliquer sur `Add > Visualization`. Chaque panneau est associé à une datasource et à une ou plusieurs requêtes. Pour les panneaux de logs, utiliser le type de visualisation Logs ; pour les graphiques de répartition, utiliser Pie chart ou Time series.
    
  Les requêtes sont écrites en LogQL.

  > Se référer à la [documentation LogQL](https://grafana.com/docs/loki/latest/query/) pour la syntaxe complète.
  
  ### Créer une variable
  Les variables permettent d'interagir dynamiquement avec les panneaux. Elles se configurent depuis :
    `Dashboard settings > Variables > New variable`
    choisir le type selon l'usage, puis référencer la variable dans les requêtes via la syntaxe `$nom_variable`.
  > La référence LogQL complète est disponible dans la [documentation officielle Grafana Loki](https://grafana.com/docs/grafana/latest/visualizations/dashboards/build-dashboards/create-dashboard/).
  
  ## 1.4 Contact points
  Les contact points peuvent être configurés via l'interface Grafana (`Alerting > Contact points`) ou provisionnés via fichier YAML.
  Les champs principaux du fichier de provisioning :
  
  | Champ | Valeur |
  |---|---|
  | `name` | Nom du contact point, référencé dans les notification policies |
  | `receivers` | Liste des destinataires associés à ce contact point |
  | `uid` | Identifiant unique, utilisé pour référencer le reciever de cecontact point |
  | `type` | Type de notification (email, slack, webhook, etc.) |
  | `settings` | Paramètres spécifiques au type choisi (adresse email, webhook URL, etc.) |

  ## 1.5 Alert Rules 
  Les alertes se créent et se configurent depuis l'interface Grafana (`Alerting > Alert rules`) ou via fichier YAML provisionné.
  Les champs clés d'une règle YAML :

  | Champ | Valeur |
  |---|---|
  | `title` | Nom de l'alerte |
  | `condition` | Référence du nœud d'évaluation final (ex : C) |
  | `data` | Définition des requêtes et expressions d'évaluation |
  | `for` | Durée pendant laquelle la condition doit être vraie avant déclenchement |
  | `interval` | Fréquence d'évaluation du groupe d'alertes |
  | `notification_settings.receiver` | Contact point destinataire |

  > Attention : il n'est pas possible d'associer une alerte créée depuis l'UI à un groupe provisionné directement. Il faut créer l'alerte, puis la déplacer manuellement dans le groupe cible.

  La référence complète est disponible dans la [documentation officielle Grafana](https://grafana.com/docs/grafana/latest/alerting/set-up/provision-alerting-resources/file-provisioning/).
# 2. Configuration de Alloy
Alloy fonctionne selon un modèle de pipeline. Les logs transitent à travers plusieurs blocs successifs, chacun assurant une étape du traitement : détection des sources, collecte, parsing et envoi.
Les sources sont soit définies statiquement via `local.file_match`, soit découvertes dynamiquement par les blocs `discovery.`. Les logs y sont ensuite collectés par les blocs `loki.source.`, traités et enrichis de labels par les blocs `loki.process`, puis transmis à Loki via `loki.write`.
  ## 2.1 Ajouter une source
  ### 2.1.1 discovery.docker
  Utilisé dans un environnement Docker, discovery.docker détecte automatiquement les conteneurs actifs sur le socket Docker et les expose comme cibles de collecte pour les autres composants.
  - `host` : socket Docker à utiliser
  - `refresh_interval` : fréquence de mise à jour de la liste des conteneurs

  ### 2.1.2 discovery.relabel
  Utilisé conjointement avec `discovery.docker`, `discovery.relabel` permet de filtrer et de transformer les cibles découvertes avant de les transmettre aux blocs suivants du pipeline.
  - `targets` : cibles auxquelles seront appliquées les règles définies dans les blocs rule
  - `rule` : bloc répétable définissant les transformations à appliquer — filtrage, renommage de labels, extraction de valeurs

  ### 2.1.3 local.file_match
  Définit statiquement les fichiers dans lesquels Alloy doit collecter les logs. À utiliser pour toute source de logs accessible via le système de fichiers (logs Apache, logs système Linux, etc.).
  - `path_targets` : liste des chemins absolus des fichiers à surveiller, accepte les wildcards (*)
  - `sync_period` : fréquence de synchronisation de la liste des fichiers
    ```
      local.file_match "apache_logs" {
        path_targets = [
          { __path__ = "/var/log/apache2/*.log" },
        ]
        sync_period = "5s"
      }
    ```
  ### 2.1.4 loki.source.
  Collecte les logs depuis les sources définies en amont et les transmet à l'étape suivante du pipeline via `forward_to`, qui est un argument obligatoire commun à tous les blocs `loki.source.`
  #### loki.source.file
  Collecte les logs depuis les fichiers définis par `local.file_match`.
  - `targets` : référence le bloc `local.file_match` correspondant
  - `forward_to` : étape suivante du pipeline
  - `tail_from_end` : si `true`, ne lit que les nouvelles entrées au démarrage
  #### loki.source.windowsevent
  Collecte les logs depuis l'observateur d'événements Windows.
  - `eventlog_name` : nom du journal Windows à surveiller (ex : System, Application)
  - `forward_to` : étape suivante du pipeline
  - `poll_interval` : fréquence de lecture des nouveaux événements

  #### loki.source.docker
  Collecte les logs des conteneurs découverts par discovery.docker.
  - `targets` : référence le bloc discovery.docker correspondant
  - `relabel_rules` : référence le bloc discovery.relabel correspondant
  - `forward_to` : étape suivante du pipeline

  ### 2.1.5 loki.process
  loki.process est le bloc de traitement du pipeline. Il reçoit les logs bruts et leur applique une série de stages successifs, chacun assurant une étape du parsing ou de l'enrichissement. Les stages s'exécutent dans l'ordre dans lequel ils sont déclarés.
  #### Stages disponibles
  - `stage.regex` : Extrait des valeurs du message de log via une expression régulière. Les groupes nommés deviennent des champs réutilisables dans les stages suivants.
    - `expression` : expression régulière avec groupes nommés (?P<nom>)

  - `stage.json` : Extrait des valeurs depuis un message de log au format JSON.
    - `expressions` : map des champs à extraire et leur clé source

  - `stage.template` : Génère une nouvelle valeur à partir de champs existants via un template Go.
    - `source` : nom du champ à créer ou modifier
    - `template` : template Go utilisant les champs extraits par les stages précédents

  - `stage.labels` : Transforme des champs extraits en labels Loki indexés.
    - `values` : map des labels à créer et leur champ source

  - `stage.static_labels` : Ajoute des labels Loki à valeur fixe, identiques pour tous les logs traités par ce bloc.
    - `values` : map des labels et leurs valeurs

  - `stage.drop` : Filtre et supprime les logs correspondant à un critère, qui ne seront pas transmis à l'étape suivante.
    - `expression` : expression régulière — supprime les logs dont le message correspond
    - `source` : champ à évaluer — supprime les logs dont ce champ correspond à value
    - `value` : valeur attendue pour le champ défini dans source

  - `stage.label_drop` : Supprime des labels précédemment définis.
    - `values` : liste des labels à supprimer

  ### 2.1.6 loki.write
  Bloc final du pipeline, `loki.write` transmet les logs traités à l'instance Loki cible.
  - `url` : endpoint d'ingestion Loki (ex : http://<IP_HOST>:3100/loki/api/v1/push)
  - `tenant_id` : identifiant d'organisation, doit correspondre au header `X-Scope-OrgID` configuré dans la datasource Grafana
